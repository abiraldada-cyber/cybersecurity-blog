Understanding OWASP Top 10

The industry standard awareness document for web application vulnerabilities.

⚠️

EDUCATIONAL LAB ENVIRONMENT ONLY

All vulnerability analysis, payloads, and scenarios detailed on this page are presented strictly for educational purposes and authorized research within academic sandboxes. Do not execute these payloads against systems you do not own or lack written permission to audit.

1. Basic: What is the OWASP Top 10?

If you are just getting started with web security, you will hear the term **OWASP** (Open Web Application Security Project) everywhere. Every few years, OWASP releases a list of the ten most critical security risks facing web applications.

This list is not just a theoretical catalog—it is compiled from analyzing thousands of real-world security reviews and data breaches. It serves as a guide for developers to avoid building security flaws into their code and helps security analysts audit applications systematically.

2. Practical: User vs. Attacker Analysis

How normal interactions contrast with malicious exploits across all ten risk areas

A01:2021 - Broken Access Control

Explanation: Restrictions on what authenticated users are allowed to do are not properly enforced. Users can access other accounts or execute administrator functions.

  • How User Uses: Views their own account dashboard at https://site.com/profile?id=102.
  • How Attacker Uses: Changes the ID parameter to view profiles belonging to other users.
  • Practical Example: Accessing https://site.com/profile?id=103 or loading an administrative path directly without a session: https://site.com/admin/delete-user.
  • Prevention: Implement strict, server-side access controls (RBAC) and validate session permissions for every request.

A02:2021 - Cryptographic Failures

Explanation: Weak or absent encryption of data in transit or at rest. This allows sensitive information (passwords, bank data, keys) to be intercepted or decoded.

  • How User Uses: Submits their credit card data, expecting it to be fully encrypted and hidden.
  • How Attacker Uses: Sniffs unencrypted HTTP traffic or extracts weakly hashed passwords from a database breach.
  • Practical Example: Sending data over http://site.com in plaintext, or using deprecated MD5 hashing for database password storage: MD5("abiral") = 8c3cd9d1ef4b81b8fa8a203922f5c71a.
  • Prevention: Force HTTPS everywhere with TLS 1.3, disable deprecated cipher algorithms, and use strong hashing algorithms like Argon2 or bcrypt.

A03:2021 - Injection (SQLi, XSS, Command Injection)

Explanation: User input is accepted and passed directly to an interpreter (like SQL or bash) without validation, allowing the input to execute as code.

  • How User Uses: Enters their login name abiral into the login username field.
  • How Attacker Uses: Enters malicious syntax to alter the structure of the database query.
  • Practical Example: Submitting ' OR '1'='1 in a login form to bypass the password check, or inserting a script payload like <script>alert('XSS')</script> in a message board.
  • Prevention: Use parameterized queries (prepared statements), enforce input whitelisting, and perform context-aware output encoding.

A04:2021 - Insecure Design

Explanation: Flaws in the application's logic and business flow, rather than implementation errors. The code might be written correctly, but the feature's design is insecure.

  • How User Uses: Recovers their account using security questions (e.g. "Which city were you born in?").
  • How Attacker Uses: Researches public records or social media posts to answer the questions and reset the target's password.
  • Practical Example: A password recovery flow that trusts a user's answer to "What is your pet's name?" without enforcing multi-factor verification (MFA).
  • Prevention: Integrate threat modeling into the early stages of software development, and avoid using weak logic like security questions for verification.

A05:2021 - Security Misconfiguration

Explanation: Servers or application frameworks configured incorrectly, such as leaving default accounts active, exposing system error logs, or keeping unnecessary ports open.

  • How User Uses: Accesses the normal application interface.
  • How Attacker Uses: Accesses unhardened administration portals using default passwords or checks error messages for server paths.
  • Practical Example: Accessing a router panel or administrative portal and logging in using admin / admin.
  • Prevention: Change all default credentials, disable directory listing, and turn off detailed system error stack traces on production servers.

A06:2021 - Vulnerable and Outdated Components

Explanation: The website uses external libraries, plug-ins, or frameworks that contain known security bugs and have not been patched.

  • How User Uses: Uses website components that depend on open-source packages (like a PDF generator or log processor).
  • How Attacker Uses: Identifies out-of-date service versions and uses public exploit code to execute commands.
  • Practical Example: Exploiiting unpatched systems using known vulnerabilities like Log4j (CVE-2021-44228) by injecting a payload string into a header input.
  • Prevention: Maintain an updated Software Bill of Materials (SBOM) and run continuous vulnerability scans on dependencies.

A07:2021 - Identification and Authentication Failures

Explanation: Session cookies or login validation systems that are poorly implemented, allowing attackers to brute-force accounts or hijack sessions.

  • How User Uses: Logins to their account with their password.
  • How Attacker Uses: Automates thousands of login attempts using list databases of common passwords, exploiting the lack of rate limits.
  • Practical Example: Sending 10,000 post requests to /login testing common passwords (e.g. password123, 123456) without the system locking the account.
  • Prevention: Enforce multi-factor authentication (MFA), implement rate-limiting, and enforce complex password policies.

A08:2021 - Software and Data Integrity Failures

Explanation: Code updates or serialized data structures are trusted and processed by the system without verifying their integrity and authenticity.

  • How User Uses: Automatically receives software patches or logs in using cookie state objects.
  • How Attacker Uses: tampers with serialized objects in cookies to gain administrator privileges.
  • Practical Example: Manipulating a base64 encoded user session cookie containing {"user":"abiral","role":"user"} and changing it to read {"user":"abiral","role":"admin"}.
  • Prevention: Sign and hash all software updates, and never trust serialized inputs from user cookies or requests without validation keys.

A09:2021 - Security Logging and Monitoring Failures

Explanation: The system does not log security-critical events (like failed logins or unauthorized access attempts), allowing attackers to operate undetected.

  • How User Uses: Interacts with the web application normally.
  • How Attacker Uses: Performs unauthorized scans or brute-force tests over days, knowing the system doesn't generate alerts.
  • Practical Example: Attacking a website and generating 5,000 server errors, but checking the administration panel to find that no logs or security alerts were recorded.
  • Prevention: Ensure all login failures and server errors are logged securely and monitored in real-time by a SIEM system.

A10:2021 - Server-Side Request Forgery (SSRF)

Explanation: A web server takes a user-supplied URL parameter and fetches it without validation, allowing the server to query internal services or local resources.

  • How User Uses: Inputs an image URL to update their profile picture: https://site.com/fetch?url=https://avatar.com/me.jpg.
  • How Attacker Uses: Changes the target URL to point to internal services or local loops.
  • Practical Example: Querying the server's local interfaces: https://site.com/fetch?url=http://127.0.0.1:8080/admin or fetching cloud metadata: http://169.254.169.254/latest/meta-data/.
  • Prevention: Disable redirects, implement URL input whitelists, and block the server from contacting internal network addresses directly.

3. Advanced: Secure Development Lifecycle

To defend against the OWASP Top 10 systematically, organizations must implement a Secure Software Development Lifecycle (SSDLC).

Instead of checking for security vulnerabilities only before deployment, security checks must be integrated into every step of the development process:

Threat Modeling

Analyze design choices early in the planning phase to find logical and business-flow security flaws before any code is written.

Static Code Audits (SAST)

Automate source code scans during code commits to locate vulnerable coding patterns before compilation.

Dynamic Testing (DAST)

Run automated vulnerability scans on the compiled, running application to simulate attacker interactions.
