CRYPTOGRAPHIC PRINCIPLES

Cryptographic Science

The mathematics of securing data integrity and confidentiality.

1. Basic: Hashing and Data Integrity

Before encrypting files, we must learn how to verify that they haven't been tampered with. This is done using **hash functions**. A hash function is a one-way mathematical formula that takes any input (like a word or a large file) and converts it into a fixed-length string of characters.

A secure hash has three critical properties:

  • One-Way: You can easily generate a hash from a file, but you cannot reverse the hash back into the original file.
  • Deterministic: The same input will always produce the exact same hash.
  • Collision Resistant: It is practically impossible to find two different inputs that produce the same output hash.

2. Practical: Symmetric vs. Asymmetric Encryption

To keep data private, we encrypt it. There are two main ways to configure encryption keys: symmetric (shared keys) and asymmetric (public-private keypairs).

Symmetric (Shared Key)

Symmetric encryption uses **one key** to both encrypt and decrypt files. It is extremely fast and suited for encrypting bulk data (like local hard drives or database files).

  • Algorithms: AES-256, ChaCha20
  • Analogy: A lockbox where the same physical key locks and unlocks the box.
  • The Flaw: If you need to send files to a friend, you must somehow share the key first. If an attacker intercepts the key, they can decrypt all your messages.

Asymmetric (Keypair)

Asymmetric encryption solves the key sharing issue using a mathematically linked **keypair**: a Public Key (anyone can see) and a Private Key (kept secret).

  • Algorithms: RSA, ECC (Elliptic Curve Cryptography)
  • Analogy: A mailbox where anyone can drop mail through the slot (Public Key), but only the owner can unlock the box to read it (Private Key).
  • How it is used: You encrypt the data with the recipient's public key. Only their private key can decrypt it.

3. Advanced: Public Key Infrastructure (PKI) & Digital Signatures

In a global network like the internet, how do you verify that a public key actually belongs to the person or site you think it does? We use **PKI**.

PKI is a network of trust that binds cryptographic keys to user identities using digital certificates and trust authorities.

Certificate Authorities

CAs are trusted organizations (like Let's Encrypt or DigiCert) that verify domains and sign digital certificates, confirming that a public key belongs to the domain (e.g., bank.com).

Digital Signatures

Provides non-repudiation. A sender hashes a file and encrypts the hash with their Private Key. The receiver decrypts it with the sender's Public Key. If the hashes match, the sender's identity is verified.

SSL/TLS Certificates

Browser trust stores contain the public keys of root CAs. When you visit an HTTPS site, the browser checks the site's certificate signature against these root keys to verify the connection.
