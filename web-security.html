WEB SECURITY LABORATORY

Web Application Security

How vulnerabilities are intercepted, tested, and resolved.

⚠️

EDUCATIONAL LAB ENVIRONMENT ONLY

All payloads, code samples, and interception examples on this page are provided strictly for authorized educational research in laboratory environments. Never attempt to intercept or manipulate traffic for systems you do not own or have explicit authorization to test.

1. Basic: How Web Applications Work

Websites are built on a simple exchange: you (the client) make a request, and the server provides a response. This communication is handled by **HTTP** (Hypertext Transfer Protocol).

When you fill out a form or search for a product, your browser packages that input into an HTTP request and sends it off. Security issues arise when the server blindly trusts that input and processes it directly in backend databases or reflects it back to other users without checking it first.

2. Practical: Traffic Interception & Exploit Labs

To audit web applications, security professionals use an intercepting proxy. **Burp Suite** is the industry-standard tool for this. It sits between your browser and the web server, allowing you to capture, read, and edit HTTP requests before they reach the server.

Using Burp Suite to Intercept Traffic

When you click "Submit" on a webpage, the request is sent immediately. With Burp Suite, you configure your browser to route traffic through Burp's local port (usually 127.0.0.1:8080).

Once "Intercept is ON" in Burp, the request freezes in transit. You can inspect headers, modify variables (like user roles or item prices), and send the modified request to the server to test how it reacts.

This is how we identify parameter tampering and injection vulnerabilities—by modifying requests directly in transit.

* Testing: An auditor might change "price=500" to "price=1" to see if the server processes payments without server-side validation.

SQL Injection (SQLi)

⚠️ Educational Lab Use Only

Normal Input:
Username: abiral | Password: 123456
Internal Mechanism: The database runs the query:
SELECT * FROM users WHERE username = 'abiral' AND password = '123456';

Malicious Input:
Username: ' OR '1'='1 | Password: (blank)
Internal Mechanism: The query becomes:
SELECT * FROM users WHERE username = '' OR '1'='1';

Why it works: The quote finishes the string input early, and the injected logic '1'='1' is always true. The database runs the logic and returns a valid user profile, bypassing the password check completely.

Cross-Site Scripting (XSS)

⚠️ Educational Lab Use Only

Normal Comment:
Input: Nice blog
Internal Mechanism: The browser renders:
<p>Nice blog</p>

Malicious Comment:
Input: <script>alert('XSS')</script>
Internal Mechanism: The browser receives:
<p><script>alert('XSS')</script></p>

Why it works: Instead of seeing plain text, the browser sees a script tag. It parses it as active code and runs the JavaScript. Attackers exploit this to capture session cookies or redirect users to phishing sites.

3. Advanced: Application Hardening

Securing web applications requires multiple layers of defense: from coding standards to perimeter filters.

By adopting modern security architectures, we can mitigate vulnerabilities even if coding mistakes occur.

Prepared Statements

Defeats SQL injection. Parameters are sent to the database separate from the query syntax. The database treats inputs strictly as values, never executing them as SQL logic.

Output Encoding & CSP

Defeats XSS. HTML entity encoding converts special characters (like < to &lt;) so they cannot form executable tags. Content Security Policies (CSP) restrict script execution sources.

Layer 7 WAFs

Web Application Firewalls inspect incoming payloads at the application layer, filtering out known injection signatures and blocking malicious requests before they reach the backend application.
